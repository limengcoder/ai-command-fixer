# AI Command Fixer 产品设计文档

日期：2026-06-11

更新：2026-06-12

## 0. 当前版本决策摘要

- 第一版承载形式确定为 Web 工具页，首屏直接提供命令粘贴、修复、核对和复制能力。
- MVP 默认采用“单命令优先模式”：用户输入默认被视为一条需要修复的长命令，不主动拆成多条命令。
- 多命令识别不再作为第一版默认体验，后续可作为显式高级模式或用户手动开启的能力。
- 复杂 shell 结构，例如 `for ... do ... done`、`if ... then ... fi`、`while ... do ... done`，必须保持为一条命令结构。
- MVP 增加“本地暂存”：默认不保存，用户主动暂存时仅保存修复后的完整命令，并按过期时间自动清理。
- skill、CLI、浏览器扩展、批量处理等承载形式暂不进入第一版范围，仅作为后续探索方向。

## 1. 背景

用户在让 AI 生成服务器部署、数据验证、脚本执行等命令时，经常遇到命令被聊天界面或 Markdown 排版折成多行的问题。尤其是 Python `-c`、SQL 查询、长参数命令、JSON 参数命令，被意外插入换行后，直接复制到服务器终端执行会触发语法错误或命令中断。

常见的“换行替换为空格”工具不能稳定解决这个问题：有些换行需要合并，有些缩进会留下多余空格，有些命令本来就有多条，需要分别复制。完全删除空格又会破坏命令语义。

本项目第一版聚焦一个明确问题：把 AI 回复中本应单行执行的命令，自动修复为可复制、可执行的单行命令。

## 2. 目标

### 2.1 产品目标

- 支持用户粘贴整段 AI 回复或单条长命令。
- 默认把输入修复为一条可核对、可复制的命令。
- 自动修复 AI 折行、反斜杠续行、换行造成的重复空格。
- 每条结果提供独立复制按钮，并支持点击结果框复制。
- 所有处理在浏览器本地完成，不上传、不登录；默认不保存命令，主动暂存除外。
- 用清晰的前后对比和提示，让用户知道工具改了什么。

### 2.2 非目标

- 不处理 `<<EOF` / here-doc / 复杂多行脚本。
- 不生成脚本文件执行版。
- 不做 base64 编码传输。
- 不做账号、云端历史、团队协作、分享链接。
- 不做后端 API、数据库或服务器存储。
- 不使用 AI 模型辅助解析命令。
- 第一版不默认自动拆分多条命令。
- 第一版不提供 CLI、Codex skill、浏览器插件或批量命令处理入口。

## 3. 用户与场景

### 3.1 目标用户

- 经常复制 AI 生成命令到服务器执行的开发者、运维、产品技术人员。
- 使用 AI 协助部署、数据检查、临时脚本、SQL 验证的人。
- 对命令隐私敏感，不希望把服务器命令上传到第三方服务的人。

### 3.2 核心场景

用户从 AI 回复中复制如下内容：

```text
验证：

/home/venvs/geo/bin/python -c "from dotenv import load_dotenv; load_dotenv('.env'); from models.database import get_db; db=get_db(); rows=db.execute_query(\"SELECT id,pipeline_code,batch_code,status,current_step,error_message
FROM geo_pipeline_runs WHERE batch_code='yilan_20260610_worker_smoke1'\"); [print(r) for r in rows]"
```

工具输出：

```bash
/home/venvs/geo/bin/python -c "from dotenv import load_dotenv; load_dotenv('.env'); from models.database import get_db; db=get_db(); rows=db.execute_query(\"SELECT id,pipeline_code,batch_code,status,current_step,error_message FROM geo_pipeline_runs WHERE batch_code='yilan_20260610_worker_smoke1'\"); [print(r) for r in rows]"
```

## 4. 协作模式

项目采用阶段性小队模式。

### 4.1 产品经理 / 项目负责人

当前主线程承担该角色。用户只与产品经理对接。产品经理负责：

- 维护 PRD 和设计文档。
- 拆解任务和验收标准。
- 记录范围、决策和进度。
- 在需要时协调设计、前端、测试、运维、增长/运营、开源选型 agent。
- 最终整合与验收。

### 4.2 阶段性角色

- 设计 Agent：在需求稳定后输出页面结构、交互状态和视觉方向。
- 前端 Agent：在设计和规则明确后实现纯前端工具。
- 测试 Agent：实现后基于真实样例、边界样例、回归样例验证工具。
- 运维 Agent：发布前提供静态部署路径、发布检查表和回滚建议。
- 增长/运营 Agent：负责开源发布材料、README、FAQ、SEO、GEO/AI-search、多语言和内容规划。
- 开源选型 Agent：按需调研 GitHub 相似项目，输出候选仓库、license、维护情况、可复用模块和推荐方案。

暂不设置后端开发、数据库或云端算法角色。

## 5. 产品范围

### 5.1 输入

- 粘贴整段 AI 回复。
- 粘贴一条普通命令。
- 粘贴一条复杂 shell 长命令。
- 多条普通命令仅作为后续显式模式处理，不作为第一版默认输入预期。

### 5.2 输出

- 默认输出一条修复结果；显式多命令模式后续再输出命令列表。每条结果包含：
  - 原始候选片段。
  - 修复后的单行命令。
  - 复制按钮。
  - 修复说明。
  - 风险或不支持提示。
- 结果框本身支持点击复制。
- 如果后续开启显式多命令模式，再提供复制全部按钮。

### 5.3 本地存储

第一版可以使用浏览器本地存储保存：

- 用户偏好，例如自动优先模式、是否展示差异。
- 自定义识别规则或命令前缀。
- UI 显示选项。
- 用户主动暂存的修复后完整命令，且必须带过期时间。

第一版默认不保存：

- 原始 AI 回复全文。
- 原始命令。
- 命令历史。
- 任何可能包含服务器路径、数据库名、token、生产信息的文本。

主动暂存限制：

- 只保存修复后的完整命令。
- 不保存原始输入或修复前文本。
- 默认过期时间 24 小时，可选 1 小时和 7 天。
- 最多保留 20 条。
- 支持单条删除和清空全部。

## 6. 核心解析方案

第一版采用状态机规则引擎，在浏览器本地运行。

### 6.1 为什么不用简单替换

简单把换行替换成空格会产生两个问题：

- 无法区分“同一条命令被折断”和“多条命令之间的换行”。
- 无法根据引号、括号、反斜杠续行判断合并边界。

### 6.2 为什么不用 AI 辅助

AI 辅助解析可能更智能，但第一版不采用：

- 命令内容可能敏感，上传会降低信任。
- 需要后端或用户 API Key。
- 成本和响应时间更高。
- 结果不可解释，不利于测试和稳定迭代。

### 6.3 状态机流程

1. 文本预处理：
   - 去掉 Markdown 代码块边界。
   - 去掉常见 shell 提示符前缀，例如 `root@host:path#`。
   - 保留用户原文用于差异展示。

2. 候选命令识别：
   - 默认模式下，不主动拆分用户输入，优先把输入视为一条长命令。
   - 如果未来开启显式多命令模式，再启用多命令候选识别。
   - 识别常见命令前缀：`cd`、`python`、`pip`、`npm`、`pnpm`、`yarn`、`node`、`git`、`docker`、`kubectl`、`curl`、`ssh`、`scp`、`rsync`、`bash`、`sh`。
   - 识别绝对路径可执行文件，例如 `/home/venvs/geo/bin/python`。
   - 识别相对路径脚本，例如 `scripts/create_pipeline_run.py` 需要依附前面的可执行命令。
   - 忽略明显说明文字，例如“验证：”“看 raw：”。

3. 行合并：
   - 如果行尾是反斜杠续行符，去掉反斜杠并与下一行合并。
   - 如果上一行存在未闭合引号、括号、方括号、花括号，尝试与下一行合并。
   - 如果下一行是缩进行且不像新命令，尝试并入当前命令。
   - 默认模式下，即使命令下一行看起来像新命令，也优先并入当前长命令。
   - 显式多命令模式中，如果检测到新命令起点，才切分为新命令。
   - `for/do/done`、`if/then/fi`、`while/do/done` 等 shell block 必须作为同一条命令处理。

4. 空白归一：
   - 将由换行和缩进造成的连续空白压成单个空格。
   - 不删除引号内必要空格。
   - 不改变普通参数之间的单个空格。

5. 输出与提示：
   - 对每条命令生成修复说明，例如“合并 1 处换行”“移除 3 处续行缩进”。
   - 如果检测到 `<<EOF`，标记为不支持的多行结构，不自动修复。
   - 如果检测到高风险命令，例如 `rm -rf`、`DROP DATABASE`，显示醒目提示，但不阻止复制。

## 7. 界面设计要求

第一版是工具型界面，不做营销落地页。首屏直接可用。

### 7.1 页面结构

- 顶部：产品名、隐私提示、本地处理说明。
- 输入区：大文本框，支持粘贴整段 AI 回复。
- 操作区：
  - 修复按钮。
  - 清空按钮。
  - 示例按钮。
  - 偏好设置入口。
- 结果区：
  - 默认展示一条修复结果。
  - 单条复制按钮。
  - 暂存按钮。
  - 点击结果框复制。
  - 自动修复点。
- 本地暂存区：
  - 默认折叠，位于主结果下方或设置区附近。
  - 展示用户主动暂存的修复后命令。
  - 支持复制、删除、清空全部和过期清理。
- 设置区：
  - 展示差异开关。
  - 自定义命令前缀。
  - 本地偏好清除按钮。

### 7.2 交互状态

- 空状态：提示用户粘贴 AI 回复或命令。
- 成功状态：显示修复结果、单条复制按钮和自动修复点。
- 无命令状态：提示未识别到命令，并建议用户手动粘贴单条命令。
- 不支持状态：检测到 here-doc 或复杂多行脚本时提示暂不支持。
- 风险状态：检测到危险命令时显示风险标识。
- 复制状态：复制成功后短暂显示反馈。
- 暂存状态：暂存成功、超过上限、删除、清空和过期清理都需要明确反馈。

## 8. 隐私与安全

- 默认不上传任何文本。
- 默认不保存命令。
- 用户主动暂存时，仅保存修复后的完整命令到当前浏览器本地，并按过期时间自动清理。
- 暂存不是历史记录，不自动保存，不云同步。
- 页面需要明确写出“所有处理均在浏览器本地完成”。
- 本地存储默认只保存偏好和自定义规则。
- 本地暂存只保存用户主动保存的修复后命令。
- 提供清除本地设置按钮。
- 提供清空本地暂存按钮。
- 对高风险命令只做提示，不做安全保证。

## 9. 测试与验收

### 9.1 样例测试

必须覆盖：

- Python `-c` 内部 SQL 被折断。
- 默认单命令模式下，复杂 shell 长命令不会被错误拆分。
- 显式多命令模式作为后续能力另行测试。
- 反斜杠续行命令合并。
- JSON 参数命令合并。
- 普通说明文字被忽略。
- `<<EOF` 被识别为不支持，不自动修复。
- `rm -rf` 等高风险命令出现提示。

### 9.2 验收标准

- 用户粘贴单条长命令后，工具能输出一条可复制单行命令。
- 修复后的 Python `-c` 命令不包含意外换行。
- `for ... do ... done` 结构不会被拆成多条命令。
- 路径、日期、SQL 标识符、字段名等高风险断点能够正确修复。
- 刷新页面后不会恢复上一条命令内容。
- 用户偏好可以在刷新后保留。
- 主动暂存的未过期命令刷新后可见；未主动暂存的命令刷新后不可见。
- 过期暂存项会自动清理。
- 断网状态下仍可使用核心功能。

## 10. 发布方式

第一版采用纯前端静态部署：

- 可部署到 GitHub Pages、Vercel、Cloudflare Pages 或客户自己的静态托管环境。
- 不需要后端服务器。
- 不需要数据库。
- 不需要环境变量。

运维 Agent 在发布前补充具体部署步骤和检查表。

## 12. 已确认决策

- 协作模式：阶段性小队模式。
- 第一版运行方式：纯前端静态工具。
- 后端：不需要。
- 第一版公开发布语言：中英并行。
- 开源策略：准备开源，默认采用 MIT License。
- 统计策略：允许接入隐私友好的匿名统计和 GSC、GA 等主流运营工具，但禁止采集命令内容、修复结果、命令 hash、路径、token、剪贴板内容或命令历史。
- 输入模式：默认单命令模式；显式多命令模式后续再做。
- 修复原则：自动优先。
- 输出：默认输出一条修复结果，支持单条复制按钮和点击结果框复制。
- 本地存储：保存偏好和自定义规则；主动本地暂存时保存修复后命令，默认不保存命令历史。
- 核心解析：状态机规则引擎。
- here-doc：第一版不处理。
