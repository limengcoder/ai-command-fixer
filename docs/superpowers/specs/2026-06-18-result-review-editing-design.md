# AI Command Fixer v0.2 结果核对与编辑设计

日期：2026-06-18

## 1. 背景

AI Command Fixer 已经能解决用户真实遇到的多类长命令折行问题，包括 SQL、路径、日期、字段名、文件名、shell block 和 CLI 长参数名断点。当前主要瓶颈不再是“能不能修”，而是“用户能不能快速确认修了哪里，并在发现少量问题时直接把结果调整到可用”。

现有结果区支持复制、暂存、点击结果框复制、自动修复点和摘要说明。但结果本身不可编辑。如果用户发现某个参数、路径、日期或业务值还需要微调，只能复制到外部编辑器修改，增加了来回切换成本。

v0.2 聚焦结果核对与轻量编辑，不扩展增长、SEO、CLI、插件、浏览器扩展或复杂语法解析。

## 2. 产品目标

- 让用户更清楚地看到自动修复了哪些位置。
- 让用户在发现少量问题时，可以直接编辑修复后的结果。
- 复制和暂存都以用户最后确认的结果为准。
- 保持当前“修复 -> 核对 -> 复制”的快速主流程。
- 不改变隐私承诺：所有处理仍在浏览器本地完成，默认不保存命令。

## 3. 非目标

- 不做完整 diff 编辑器。
- 不做语法树级 SQL、shell 或 Python 解析。
- 不判断命令语义是否正确或执行是否安全。
- 不做编辑历史、撤销栈、版本对比或自动保存草稿。
- 不把编辑内容上传、写入 URL、写入统计事件或自动保存到本地历史。
- 不在用户手动编辑后自动重新运行修复器。
- 不默认把结果区变成可编辑状态。

## 4. 核心决策

### 4.1 结果编辑模式

默认状态下，结果框保持只读：

- 点击结果框仍然复制当前结果。
- 结果卡片保留单条 `复制` 按钮。
- 结果卡片保留 `暂存` 按钮。
- 新增低权重 `编辑` 按钮。

点击 `编辑` 后进入编辑模式：

- 当前修复结果替换为可编辑文本框。
- 卡片显示 `已手动编辑` 标记。
- 操作按钮变为：
  - `复制编辑结果`
  - `暂存编辑结果`
  - `恢复自动修复结果`
- 编辑模式下，点击文本框只负责定位光标，不触发复制。

### 4.2 编辑后的内容优先级

用户已确认：编辑后的结果是后续操作的唯一有效结果。

进入编辑模式后：

- `复制编辑结果` 复制用户当前编辑内容。
- `暂存编辑结果` 暂存用户当前编辑内容。
- 如果未来出现多条结果并支持复制全部，复制全部应使用每条结果的当前有效内容：已编辑则用编辑内容，未编辑则用自动修复结果。

### 4.3 恢复自动修复结果

`恢复自动修复结果` 用于放弃手动编辑：

- 文本框内容恢复为该条命令的自动修复结果。
- `已手动编辑` 标记移除。
- 卡片回到只读状态。
- 复制和暂存重新使用自动修复结果。

恢复操作不需要二次确认，因为它只影响当前页面内的临时编辑内容，不会删除本地暂存项。

## 5. 信息架构

每条结果卡片建议分为四个区域：

1. 标题与状态
   - `命令 1`
   - `可复制` / `不支持`
   - `高风险`
   - `已手动编辑`

2. 操作区
   - 只读状态：`复制`、`暂存`、`编辑`
   - 编辑状态：`复制编辑结果`、`暂存编辑结果`、`恢复自动修复结果`

3. 命令内容区
   - 只读状态：代码块，可点击复制。
   - 编辑状态：等宽字体文本框，可多行横向滚动或自动换行，但不得压缩用户输入。

4. 核对区
   - 自动修复点。
   - 修复摘要。
   - 风险提示。
   - 不支持提示。

## 6. 修复核对增强

### 6.1 修复点分组

自动修复点按理解成本分组：

- `自动修复`
  - 日期断点，例如 `2026- ⏎ 06-18 -> 2026-06-18`
  - 路径断点，例如 `/bin/ ⏎ python -> /bin/python`
  - CLI 参数名断点，例如 `--status- ⏎ concurrency -> --status-concurrency`

- `建议核对`
  - 字段名、文件名、SQL 标识符或推测类 token 修复。
  - 例如 `citation_l ⏎ ist -> citation_list`

- `暂不支持 / 风险提示`
  - here-doc、多行脚本或高风险命令提示。
  - 仍然不阻止复制，但需要明确提示工具不保证命令安全。

### 6.2 修复点说明文案

每个修复点应尽量包含：

- 类型：日期断点、路径断点、命令参数名断点、字段名断点、SQL 片段、推测修复。
- 原始片段。
- 修复后片段。
- 一句简短说明。

示例：

```text
命令参数名断点
--status- ⏎ concurrency -> --status-concurrency
说明：识别为 CLI 长选项名被换行打断，已合并连字符两侧内容。
```

```text
SQL 关键字边界
FROM ⏎ geo_raw_responses -> FROM geo_raw_responses
说明：保留 SQL 关键字和表名之间的必要空格，避免粘连为 FROMgeo_raw_responses。
```

### 6.3 展示优先级

- 结果命令仍然是页面核心，不能被核对区挤到很远。
- 默认展示精简核对信息。
- 若修复点很多，允许折叠为“查看全部修复点”。
- 风险提示优先级高于普通修复点，应放在结果命令下方醒目位置。

## 7. 数据与状态

每条命令结果需要区分两个值：

- `autoFixedText`：解析器自动修复后的结果。
- `currentText`：当前有效结果。默认等于 `autoFixedText`；用户编辑后等于编辑框内容。

每条命令结果需要状态：

- `isEditing`：是否正在编辑。
- `isManuallyEdited`：当前有效结果是否偏离自动修复结果。

操作行为：

- 复制：读取 `currentText`。
- 暂存：读取 `currentText`。
- 结果框点击复制：只在非编辑状态生效，读取 `currentText`。
- 恢复自动修复：将 `currentText` 重置为 `autoFixedText`，退出编辑状态。

不支持命令：

- 不支持项默认不提供暂存。
- 是否允许编辑不支持项由前端和产品共同评估。v0.2 推荐暂不允许编辑不支持项，避免用户误以为 here-doc 或多行脚本已被安全处理。

## 8. 隐私边界

结果编辑不改变隐私承诺：

- 编辑内容只存在当前浏览器页面状态中。
- 默认不保存编辑内容。
- 只有用户主动点击 `暂存编辑结果` 时，编辑后的完整命令才会进入本地暂存。
- 编辑内容不得进入 URL、analytics、日志、错误上报或远程请求。
- 统计事件如果后续接入，只能记录枚举行为，例如 `result_edited=true`，不能包含命令内容、编辑片段、编辑前后差异或命令 hash。

## 9. 验收标准

- 默认结果区仍然可点击复制，不进入编辑。
- 点击 `编辑` 后，结果变成可编辑文本框。
- 编辑文本框内容后，`复制编辑结果` 复制编辑后的内容。
- 编辑文本框内容后，`暂存编辑结果` 暂存编辑后的内容。
- 编辑后卡片显示 `已手动编辑`。
- 点击 `恢复自动修复结果` 后，内容回到自动修复结果，编辑标记消失。
- 编辑模式下点击文本框不会触发复制。
- 只读模式下点击结果框仍然复制当前有效结果。
- 本地暂存不保存原始输入、修复点详情、编辑前后差异或历史版本。
- `npm test` 通过。
- 浏览器人工验收覆盖桌面和窄屏。

## 10. Agent 任务拆分

### 设计 agent

- 输出结果卡片在只读和编辑状态下的交互结构。
- 设计修复点分组展示。
- 确认按钮层级：复制优先，编辑和暂存为次级操作。
- 确保移动端下编辑框、按钮和修复点不互相遮挡。

### 前端开发 agent

- 在结果卡片中实现显式编辑模式。
- 调整复制、点击结果框复制、暂存和复制全部，使其读取当前有效结果。
- 增加 `已手动编辑` 状态显示。
- 增强修复点展示，至少支持 `option` 类型文案。
- 更新静态资源 query version，避免线上缓存旧脚本。

### 测试 agent

- 增加复制/暂存读取编辑后内容的自动化或单元测试。
- 增加编辑后恢复自动修复结果的测试。
- 增加人工验收清单：只读点击复制、编辑后复制、编辑后暂存、恢复自动修复。
- 回归现有解析器测试，确保不影响最近修复的 CLI 长参数名断点。

### 运维 agent

- 实现完成并验收后，提交、推送并检查 GitHub Pages 返回状态。

## 11. 后续延期事项

- 完整差异视图。
- 编辑历史和撤销栈。
- 批量编辑多条命令。
- 重新运行解析器并保留用户编辑。
- 支持编辑 here-doc 或复杂多行脚本。
- 把编辑能力扩展成 CLI、浏览器插件或 Codex skill。
