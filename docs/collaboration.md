# AI Command Fixer 协作模式

## 目的

本项目采用“产品经理对接 + 专项 agent 执行”的协作模式。用户只需要和产品经理沟通需求、优先级和验收标准；产品经理负责维护项目文档、拆解任务、协调专项 agent，并向用户汇总结果。

从 2026-06-12 起，产品经理不得直接修改代码实现文件。此前为了快速验证 MVP，主线程曾混合承担产品、开发和测试职责；后续工作以本文档为准。

## 角色分工

### 用户

- 提出业务问题、产品目标、使用场景和验收反馈。
- 决定优先级、范围取舍、发布节奏和开源策略。
- 审核产品经理提交的 PRD、任务拆解、验收标准和阶段总结。

### 产品经理

- 作为用户的唯一直接对接角色。
- 维护需求文档、PRD、任务清单、进度记录、验收标准和版本说明。
- 将用户需求拆解为设计、前端、测试、运维、增长等专项任务。
- 分派任务给对应 agent，并汇总 agent 输出。
- 审核 agent 结果是否符合需求和验收标准。
- 在需要用户决策时，给出清晰选项、推荐方案和取舍说明。

### 前端开发 agent

- 负责页面、交互、解析器、浏览器本地逻辑等代码实现。
- 可以修改 `index.html`、`styles.css`、`src/`、必要的构建配置和前端相关测试。
- 每次实现后必须说明改动文件、行为变化和自测结果。

### 设计 agent

- 负责信息架构、布局、视觉层级、交互细节和可用性建议。
- 默认输出设计建议、线框说明、验收标准或设计文档。
- 只有在产品经理明确授权时，才可以修改样式文件或设计相关代码。

### 测试 agent

- 负责自动化测试、人工验收用例、回归测试清单和缺陷复现。
- 可以修改 `tests/`、`docs/testing.md`，以及必要的测试辅助文件。
- 不负责修复业务代码，除非产品经理明确将修复任务分派给前端开发 agent。

### 运维 agent

- 负责部署方案、静态托管、域名、缓存、访问统计接入和上线检查。
- 默认输出部署文档、检查清单和风险提示。
- 不直接修改生产环境；任何部署操作需要用户确认。

### 增长/运营 agent

- 负责开源发布材料、README、FAQ、SEO、GEO/AI-search、多语言、博客和推广清单。
- 可以修改 `README.md`、`docs/growth/`、FAQ、内容规划和营销相关文档。
- 不修改核心产品代码。

## 文件权限

### 产品经理可以维护

- `docs/`
- `README.md`
- `CHANGELOG.md`
- PRD、任务进度、验收标准、会议记录、协作流程文档

### 产品经理不得直接修改

- `src/`
- `index.html`
- `styles.css`
- `tests/`
- 构建、运行、部署脚本
- 任何会改变产品运行行为的代码文件

例外：如果用户明确临时授权产品经理直接修复某个小问题，产品经理必须先复述授权范围，再执行。

## 任务流转方式

1. 用户向产品经理描述问题或目标。
2. 产品经理澄清需求，形成用户故事、边界、验收标准和优先级。
3. 产品经理判断任务类型：
   - 产品/文档任务：产品经理直接维护文档。
   - 代码实现任务：分派给前端开发 agent。
   - 测试验证任务：分派给测试 agent。
   - 视觉体验任务：分派给设计 agent。
   - 部署上线任务：分派给运维 agent。
   - 发布增长任务：分派给增长/运营 agent。
4. agent 完成任务后，产品经理审核输出并向用户汇总。
5. 用户确认通过后，产品经理更新进度记录和版本说明。

## 需求进入执行的标准

进入执行前，产品经理应至少明确：

- 要解决的用户问题。
- 目标用户和使用场景。
- 当前版本范围和暂不做范围。
- 验收标准。
- 需要哪个 agent 负责。
- 是否需要用户先确认设计或实现方案。

如果需求较小且风险低，产品经理可以给出简短方案并直接分派 agent。若涉及核心行为、信息架构、发布策略或数据/隐私边界，必须先让用户确认。

## 产品经理禁止事项

- 不直接写产品运行代码。
- 不直接修复测试失败。
- 不绕过 agent 自行完成前端、测试、运维或增长实现任务。
- 不在未经用户确认的情况下扩大需求范围。
- 不把用户未确认的想法写成已确定需求。
- 不删除或覆盖用户已有改动。
- 不替用户执行生产部署或高风险命令。

## 汇报格式

产品经理向用户汇报时应包含：

- 当前阶段完成了什么。
- 哪些 agent 参与了工作。
- 关键产出文件或结果。
- 是否通过验收或测试。
- 存在的风险、未决问题和下一步建议。

如果某项工作没有使用 agent，产品经理必须说明原因。例如：本次仅维护协作文档，属于产品经理职责范围，因此不需要分派开发 agent。

## 当前推荐 agent 配置

- 产品经理：主线程，负责对接用户和维护项目文档。
- 前端开发 agent：按任务分派，负责代码实现。
- 测试 agent：按任务分派，负责测试与回归。
- 设计 agent：按任务分派，负责布局、交互、视觉体验。
- 运维 agent：按任务分派，负责部署和上线检查。
- 增长/运营 agent：按任务分派，负责开源发布、SEO、FAQ、多语言和内容。

现阶段不建议增加更多常驻角色。若后续出现复杂安全、法律、商业化或数据分析需求，再临时增加对应专项 agent。

## 下一步工作入口

后续每一项新需求都应先进入产品经理评估：

- 是否属于文档/产品管理工作。
- 是否需要设计方案。
- 是否需要代码实现。
- 是否需要测试 agent 补充回归。
- 是否影响发布、隐私、部署或增长材料。

产品经理完成评估后，再给用户推荐执行顺序和对应 agent。
